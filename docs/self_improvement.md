# Self-Improvement Pipeline (Iterative Offline SFT)

A sequential pipeline for iteratively improving models on low-resource hardware (single GPU). Each phase exclusively owns the GPU — no concurrent loading required.

## Concept

Standard SFT trains on FIXED expert data. Self-improvement trains on data the model GENERATES ITSELF, filtered by quality. This creates a compound improvement loop:

```
┌─────────────┐     ┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│  GENERATE   │────▶│    SCORE     │────▶│   PREPARE    │────▶│    TRAIN     │
│  (vLLM)     │     │ (judge/exec) │     │  (CPU only)  │     │ (our loop)   │
└─────────────┘     └──────────────┘     └──────────────┘     └──────────────┘
       ▲                                                              │
       └──────────────────────────────────────────────────────────────┘
                              ITERATE (2-5 rounds)
```

## Why It Works

From research (arxiv:2510.18814, "Surprising Effectiveness of Self-Tuning"):
- OSFT matches GRPO performance without reward models
- On-policy data (generated by the model itself) has better distributional alignment
- Each iteration, the model generates better data → trains on better data → improves

From DART (arxiv:2605.26924, ICLR 2026):
- External expert data can be distributionally MISMATCHED with the target model
- Self-generated data is inherently well-matched (on-policy)
- Verification ensures correctness without needing a teacher

## Memory Requirements

Each phase runs ALONE on the GPU:

| Phase | GPU Memory | Time |
|-------|-----------|------|
| Generate (vLLM, 8B model, AWQ 4-bit) | ~6 GB | Fast (batched inference) |
| Score (judge model, 8B, AWQ 4-bit) | ~6 GB | Fast |
| Score (code execution) | 0 GB (CPU) | Variable |
| Prepare (filtering + selection) | 0 GB (CPU) | Seconds |
| Train (full bf16 + optimizer) | 40-80 GB | Normal training time |

For a 24GB GPU: generate in 4-bit quantized, train on smaller model or with FSDP.
For 80GB A100: all phases fit comfortably.

## Phases in Detail

### Phase 1: Generate

Load the current model (or its quantized version) via vLLM/SGLang for fast batched generation.

```bash
# Generate N completions per prompt
python -m vllm.entrypoints.openai.api_server \
    --model ./checkpoints/current \
    --quantization awq \
    --max-model-len 8192

# Then use our generation script:
pgs generate \
    --prompts training_prompts.jsonl \
    --endpoint http://localhost:8000 \
    --n_completions 8 \
    --temperature 0.7 \
    --output generations/round_1.jsonl
```

### Phase 2: Score

Three scoring modes (can combine):

**A. Code Execution (for code tasks)**
```bash
pgs score \
    --generations generations/round_1.jsonl \
    --method execute \
    --sandbox docker \
    --output scored/round_1.jsonl
```

**B. Answer Verification (for math/factual)**
```bash
pgs score \
    --generations generations/round_1.jsonl \
    --method verify \
    --answers ground_truth.jsonl \
    --output scored/round_1.jsonl
```

**C. Judge Model (for open-ended/quality)**
```bash
# Load a judge model (can be same model or stronger)
pgs score \
    --generations generations/round_1.jsonl \
    --method judge \
    --judge_model Qwen/Qwen3-8B-Instruct \
    --output scored/round_1.jsonl
```

### Phase 3: Prepare

Filter and select best completions using our difficulty-aware pipeline:

```bash
pgs prepare \
    --model ./checkpoints/current \
    --data scored/round_1.jsonl \
    --output prepared/round_1/ \
    --strategy optimal \
    --budget 5000
```

### Phase 4: Train

Standard training on self-generated filtered data:

```bash
pgs train \
    --config configs/self_improve.yaml \
    --data.dataset prepared/round_1/scored_data.jsonl
```

## Iteration Strategy

**Round 1**: Generate from base model → score → filter → train (initial self-improvement)
**Round 2**: Generate from round-1 model → score → filter → train (compound gains)
**Round 3+**: Diminishing returns typically after 3-5 rounds

Key decisions per round:
- **Temperature**: Start high (0.8-1.0) for diversity, decrease per round (0.5-0.7) for quality
- **N completions**: More at start (8-16), fewer later (4-8) as quality improves
- **Budget**: Keep constant or increase slightly per round
- **Verification threshold**: Can tighten per round (accept only pass@1 eventually)

## Verification Methods

### For Code Agents
- Execute generated code/commands
- Check exit code, output correctness, test passage
- Compare against expected files/state
- Zero GPU memory required

### For Math/Reasoning
- Extract final answer, compare against ground truth
- Supports numeric comparison (±ε), symbolic (sympy), exact match
- Zero GPU memory required

### For General Quality (Open-ended)
- Load judge model in vLLM (quantized)
- Prompt: "Rate this response 1-5 on correctness, helpfulness, format"
- Or: pairwise comparison against a reference response
- Requires GPU but sequential (not concurrent with training)

### For Agentic Trajectories
- Execute the full trajectory in a sandbox
- Score by: task completion, efficiency (fewer turns = better), error recovery
- The ECHO paper's insight: even FAILED trajectories contain learning signal
- Include partial successes with lower weight

## Composing with Turn Scaling

When training on self-generated multi-turn traces, use `data.turn_scaling: progressive` to emphasize later turns (where the model does the hard work):

```yaml
data:
  turn_scaling: progressive      # later turns = more loss weight
  include_observations: true     # ECHO: train on tool outputs too
plugins:
  deft: true                     # adaptive loss per token
  sym_noise: true                # regularization
train:
  adagc: true                    # spike prevention
  ema: true                      # generalization
```

## Anti-Collapse Safeguards

Self-improvement can collapse if the model only generates easy/repetitive data:

1. **Diversity check**: Monitor unique n-grams in generations. If declining → increase temperature
2. **Entropy monitoring**: Track `health/gns` and policy entropy. If collapsing → stop iterating
3. **Mix with original data**: Always include 20-30% of original expert data in training mix
4. **DEFT + SymNoise**: Automatic regularization against mode collapse

## Comparison with Alternatives

| Method | Requires | GPU Memory | Quality |
|--------|----------|-----------|---------|
| Standard SFT | Expert data | Training only | Baseline |
| Self-improvement (ours) | Prompts + verifier | Sequential phases | +5-15% |
| GRPO/PPO | Reward model + on-policy rollouts | 2-4× training | +10-20% |
| Expert distillation | Stronger teacher model | Teacher inference + training | +5-10% |

Self-improvement is the best trade-off for low-resource: no reward model needed, no concurrent memory, matches or approaches GRPO quality on verified tasks.
